# 标注在模型生产全周期中的参与：正式调研报告（v3.3）

## 摘要

本报告围绕标注在模型生产全周期中的参与方式展开，重点分析标注数据如何进入数据准备、训练、后训练、评测、安全和反馈迭代环节，以及不同标注数据如何支持模型能力建设。报告不做实证分析，不展开算法细节，重点梳理流程、数据、质量监控和公开实践材料中的可观察变化。

前沿实验室、评测机构和标准组织的公开材料将作为实践样例穿插在相关章节中，用于说明标注、后训练、人工评测、红队测试、数据文档化和质量控制在模型生产中的作用。公开材料不能还原机构内部完整训练数据、标注规范、样本权重和供应链组织，因此本报告只做公开材料层面的行业观察，不做公司横向评测。

## 0. 报告范围说明

### 0.1 研究对象与基本范围

本报告讨论“标注在模型生产全周期中的参与方式”。这里的模型不只指通用大语言模型，也包括垂类大语言模型、分类模型、推荐和排序模型、搜索模型、视觉模型、语音模型、NLP 模型、多模态模型，以及企业生产环境中常见的其他 AI / 机器学习模型。由于近两年前沿公开材料中，后训练、偏好数据、安全评测、红队和模型行为规范主要围绕 LLM 展开，报告会较多使用 LLM 案例；但本报告的基本框架并不只适用于 LLM。

本报告中的“标注”也不只包括传统意义上的分类、框选、分割、抽取和转写。它还包括数据质量标记、来源标记、领域标记、敏感性标记、指令样例、标准答案、改写样本、偏好排序、人工打分、A/B 选择、安全样本、拒答样本、风险标签、红队样本、人工评测、错误归因、线上反馈整理，以及领域术语、事实、规则、流程和证据来源等专家标注。

因此，标注可以理解为一种“把人类判断、业务规则、领域知识和风险边界转化为模型可使用材料”的过程。这个过程可能发生在训练前，也可能发生在后训练、评测、安全审查、上线监控和版本迭代中。它的产物可能是训练数据，也可能是评测集、红队用例、质量报告、风险标签、反馈样本、数据卡或模型卡中的说明。

### 0.2 范围边界与公开材料使用原则

本报告以公开权威材料和既有研究为依据，围绕模型生产流程、标注数据类型、数据质量控制与前沿公开实践进行归纳。报告不包含模型实验、算法复现、供应商报价、标注平台横向评测或采购判断；涉及 pre-training、mid-training、post-training、RLHF、DPO、RLAIF、LLM-as-Judge 等概念时，仅说明其与标注数据、偏好数据、评测数据和质量控制的关系。

公开材料的使用原则是按生产环节和论证主题归类，而不是按机构建立固定标签。本报告不会预设某个机构代表某种固定路线，而是根据公开材料实际涉及的生产环节，将其归入行为规范、后训练、评测、红队、数据文档化、标准化等主题中讨论；具体材料索引见附录 A。

## 1. 模型生产阶段与标注参与位置

模型生产中的标注参与可以分为阶段性位置、数据产物和影响方式三个层面。不同阶段的标注不只是人力投入多少的差异，更体现为是否定义模型行为、是否支持模型能力、是否影响上线验收以及是否进入后续版本迭代。

### 1.1 分阶段模型生产流程

模型生产流程可以按以下阶段理解：

```text
任务目标与成功标准定义
↓
数据规划与准备
↓
基础能力训练 / 预训练
↓
领域或能力增强 / Mid-training / 持续训练
↓
任务适配与行为校准 / Post-training / 后训练
↓
评测、安全审查与上线验收
↓
部署上线与运行监控
↓
反馈回流与版本迭代
```

这一流程并不意味着所有模型都必须严格经历每个阶段。传统分类模型、排序模型、搜索模型和视觉检测模型通常不会使用“post-training”这样的术语；但它们同样需要目标定义、样本选择、标签设计、训练 / 验证划分、评测验收、上线监控和错误回流。LLM 的公开材料放大了这些环节：后训练需要指令数据、示范回答、偏好数据和安全数据；上线前需要 system card / model card、安全评测、红队和第三方测试；上线后需要用户反馈、失败案例和政策更新回流。

> **实践样例：前沿模型发布材料中的阶段化生产**  
> 具体材料包括 OpenAI《[GPT-4o System Card](https://openai.com/index/gpt-4o-system-card/)》（2024-08-08）、Anthropic《[Claude Opus 4 & Claude Sonnet 4 System Card](https://www.anthropic.com/claude-4-system-card)》（2025-05，含后续修订记录）以及 Google DeepMind 的 [model cards](https://deepmind.google/models/model-cards/) 页面。GPT-4o System Card 同时说明模型数据与训练、安全评测、第三方评估和部署相关限制；Claude 4 System Card 同时覆盖训练特征、预部署安全测试、违反使用政策行为测试、agentic safety、reward hacking 等评估；Google DeepMind 的模型卡页面按模型列出发布日期或更新日期，并说明模型卡用于提供设计与评估概览。  
> 对应生产阶段：预训练 / 后训练、上线前安全评测、发布说明、上线后更新。  
> 能说明的判断：公开发布材料已经把训练、评测、部署决策和风险披露放在同一条链路里，标注数据的作用也随之从“训练输入”扩展到评测样本、安全测试、使用边界和反馈追溯。  
> 边界：这些材料不会公开完整训练集、标注规范、样本权重和内部供应链，因此只能说明阶段化生产和质量披露的存在，不能据此还原具体生产流程或比较机构优劣。

### 1.2 标注在不同阶段的参与方式

标注参与方式可以分为三类：核心塑形型参与、能力支撑型参与和治理辅助型参与。三类参与方式的差异不在于人力多少，也不在于数据量大小，而在于是否定义模型行为、是否进入训练或后训练核心数据、是否影响上线判断，以及是否决定后续版本迭代的方向。

| 模型生产阶段 | 标注参与方式 | 主要标注产物 | 参与类型 |
|---|---|---|---|
| 任务目标与成功标准定义 | 把业务目标转成可判断的标签、规则、成功标准和失败边界 | 标签体系、评分口径、拒答 / 可答边界、质量标准 | 核心塑形型参与 |
| 数据规划与准备 | 选择数据来源，标记来源、领域、语种、质量、敏感性、授权状态 | 数据治理标签、清洗记录、来源记录、样本分层 | 能力支撑型参与 / 治理辅助型参与 |
| 基础能力训练 / 预训练 | 对大规模数据做过滤、去重、质量分层、风险识别和可用性标记 | 质量标记、风险标签、去重和过滤结果 | 治理辅助型参与 |
| 领域或能力增强 / Mid-training / 持续训练 | 为领域知识、专业术语、业务流程、工具调用和高价值样本提供结构化材料 | 领域语料、知识片段、流程样例、专家标注 | 能力支撑型参与 |
| 任务适配与行为校准 / 后训练 | 编写指令、标准答案、偏好排序、安全拒答样本和行为规范样例 | SFT 数据、偏好数据、安全数据、对齐数据 | 核心塑形型参与 |
| 评测、安全审查与上线验收 | 组织人工评测、红队、回归测试、安全分类和专家审核 | 评测集、红队样本、风险分类、验收报告 | 核心塑形型参与 |
| 部署上线与运行监控 | 标记线上失败、用户投诉、风险输出、场景漂移和异常样本 | 线上反馈样本、失败案例、风险日志 | 能力支撑型参与 |
| 反馈回流与版本迭代 | 对错误样本归因，更新标签规则、评测集、训练样本和安全样本 | 错误归因、返工样本、版本化评测集 | 核心塑形型参与 / 能力支撑型参与 |

在任务目标定义阶段，标注参与的重点不是“先生产多少数据”，而是把目标变成可判断的口径。例如客服模型不是泛泛地“回答得好”，而是需要明确退款、投诉、升级工单、隐私信息、敏感承诺等场景中什么叫正确、什么叫危险、什么需要转人工。视觉检测模型也不是只需要画框，而是要界定目标类别、遮挡情况、小目标边界、是否包含背景物、重复目标如何处理。

在后训练阶段，标注的作用更加直接。示范回答、偏好排序、安全样本、规则原则和错误样本，都会影响模型如何回答、如何拒答、如何排序输出以及如何处理复杂指令。无论具体训练方法如何变化，人的判断都以示范、排序、打分、规则或反馈的形式参与模型行为的塑形。

在评测和安全审查阶段，标注往往从“训练材料”转为“验收材料”。这里的标注数据可能不是用来训练模型，而是用来证明模型是否达到上线门槛，或暴露尚未解决的风险。评测集、红队样本、专家判断和回归测试样本，都会影响模型是否能进入生产环境以及后续版本是否需要返工。

### 1.3 不同模型在同一生产阶段中的差异提示

不同模型共享“目标定义、数据准备、训练、评测、上线、反馈”的基本流程，但标注对象和质量标准存在差异。模型类型只有在影响标注对象、标注数据类型、质量标准、评测口径、专家参与程度、安全和合规边界时，才需要单独区分。

| 模型类型 | 标注重点差异 | 质量差异 |
|---|---|---|
| 通用 LLM | 指令、示范回答、偏好排序、安全拒答、红队样本、开放式人工评测 | 主观性强，需要多轮对话、价值边界和安全口径 |
| 垂类 LLM | 领域术语、专业事实、业务流程、证据来源、专家判断 | 专家参与更重，事实边界和合规边界更重要 |
| 分类 / 抽取模型 | 类别定义、实体边界、关系定义、异常样本 | 标签一致性和边界案例决定模型可靠性 |
| 推荐 / 排序模型 | 点击、转化、偏好、负反馈、曝光上下文 | 数据偏差、位置偏差和反馈延迟影响评估 |
| 搜索模型 | 查询意图、相关性、证据匹配、结果质量 | 相关性分级和 query 分布覆盖很关键 |
| 视觉模型 | 框选、分割、关键点、遮挡、小目标、场景标签 | 几何精度、漏标和多目标边界更突出 |
| 语音模型 | 转写、说话人、时间戳、噪声、口音、情绪 | 转写一致性、时间对齐和低资源语言覆盖更突出 |
| 多模态模型 | 图文对齐、视频文本对齐、跨模态证据、视觉安全 | 对齐关系和跨模态风险判断更复杂 |

例如，通用 LLM 在后训练阶段更关注通用指令、偏好和安全数据；垂类 LLM 更关注领域术语、专业事实、业务流程和专家判断；视觉检测模型更关注框选边界和场景覆盖；推荐模型更关注反馈信号是否代表真实偏好。它们共享“目标定义、数据准备、训练、评测、上线、反馈”的大流程，但标注对象和质量标准并不相同。

## 2. 标注数据如何支持模型能力建设

本章回答的问题是：不同标注数据到底支持模型什么能力？它们只服务单次训练，还是可以沉淀为后续能力建设材料？

### 2.1 标注数据的形成：载体、动作与产物

标注数据的原始载体包括文本、图像、音频、视频、日志和结构化数据。不同载体上的标注动作不同，但最终都需要转化为可训练、可评测、可追溯的数据材料。

| 标注动作类别 | 覆盖内容 | 典型产物 |
|---|---|---|
| 判断类 | 分类、打分、排序、审核、风险判断 | 类别标签、评分、偏好对、风险等级 |
| 定位 / 抽取类 | 框选、分割、实体抽取、关系抽取、时间片段定位 | 目标框、mask、实体边界、关系三元组、时间戳 |
| 转写 / 结构化类 | 语音转写、OCR 校正、字段整理、表格结构化 | 转写文本、校正文档、字段表、结构化记录 |
| 关联 / 对齐类 | 图文匹配、证据对齐、问答来源绑定、视频文本对齐 | 对齐关系、引用来源、片段映射 |
| 生成 / 改写类 | 编写指令、标准答案、改写样本、示范回答 | 指令数据、SFT 样本、标准答案、风格改写 |
| 评测 / 反馈类 | 人工评测、错误归因、用户反馈整理、失败案例标注 | 评测集、错误库、反馈样本、回归测试集 |

这些动作共同指向一个事实：标注不是单一生产动作，而是把“原始材料”转成“可用于模型生产的判断材料”。同一条原始数据也可能被多次标注。一次客服对话可以先被标记为投诉类，然后抽取订单号和诉求，再由专家判断回答是否合规，最后在上线后被标为失败案例并进入回归测试集。一次医疗问答可以先做实体抽取，再做证据来源绑定，再由专家判断是否超出知识边界。

### 2.2 关键标注数据类型及其支持的模型能力

标注数据支持模型能力的方式可以概括为七类。这里的“能力”不是抽象口号，而是模型在训练、评测、上线和迭代中被要求稳定表现的行为。

| 标注数据类型 | 主要内容 | 支持能力 |
|---|---|---|
| 数据治理 / 质量标记数据 | 来源、语种、领域、质量等级、敏感性、可用性 | 数据筛选、风险过滤、训练数据治理 |
| 任务标签数据 | 分类、实体、关系、目标框、排序标签 | 分类、识别、抽取、检测、排序 |
| 生成示范数据 | 指令、标准答案、改写样本、示范回答 | 生成、问答、任务执行 |
| 偏好数据 | 回答排序、A/B 选择、人工打分 | 偏好对齐、输出排序、用户体验 |
| 安全数据 | 风险标签、拒答样本、违规样本、红队样本 | 安全控制、拒答能力、风险识别 |
| 评测与反馈数据 | 人工评测、错误样本、用户反馈、回归测试样本 | 上线验收、版本对比、持续迭代 |
| 领域知识数据 | 术语、事实、规则、流程、专家判断、证据来源 | 垂类能力、专业能力、知识边界 |

数据治理和质量标记数据主要解决“哪些数据能进入生产流程”的问题。它们通常不直接对应模型某一项可见能力，却会影响训练数据筛选、风险过滤、低质量样本排除、敏感内容处理和后续追溯。对于大规模模型而言，数据治理本身已经成为模型生产材料的一部分。

任务标签数据是传统监督学习中最容易理解的标注类型。分类模型需要类别标签，抽取模型需要实体和关系，视觉检测模型需要目标框或分割结果，搜索和排序模型需要相关性或偏好信号。这类数据通常直接决定模型能否学习到稳定的输入输出关系。

生成示范数据、偏好数据、安全数据和评测反馈数据，是 LLM 后训练和上线判断中最典型的扩展型标注数据。这里的标注不只是“给答案打分”，而是参与塑造模型回答方式、推理呈现、拒答边界、风险识别和用户体验。

> **实践样例：后训练中的示范、偏好、安全和规则数据**  
> 具体材料包括 Ouyang et al.《[Training language models to follow instructions with human feedback](https://arxiv.org/abs/2203.02155)》（arXiv 2022）、DeepSeek-AI《[DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning](https://arxiv.org/abs/2501.12948)》（arXiv 2025）和 Alibaba Qwen Team《[Qwen3 Technical Report](https://arxiv.org/abs/2505.09388)》（arXiv 2025）。InstructGPT 论文明确使用人工示范回答和模型输出排序数据；DeepSeek-R1 报告公开描述了 cold-start data、多阶段训练和强化学习在推理能力形成中的作用；Qwen3 技术报告把 thinking / non-thinking 模式、评测和训练流程放在同一模型族报告中说明。  
> 对应生产阶段：post-training，以及部分 reasoning 能力强化阶段。  
> 能说明的判断：后训练所需的标注材料并不限于“标签”，而是包括示范、排序、偏好、规则、安全样本和推理相关样本；这些材料直接影响回答方式、拒答边界、推理呈现和任务服从。  
> 边界：这些论文和技术报告不等同于各机构完整生产实践。它们能证明公开可见的后训练数据形态和方法方向，不能推断真实生产中的数据规模、样本来源比例、人工与自动化分工或具体标注成本。

### 2.3 数据格式、交付物与使用边界

格式本身不是标注数据的核心。文本和对话数据常见为 CSV、JSON、JSONL、Parquet；图像检测常见为 COCO、YOLO 等；语音和视频数据通常需要原始文件、时间戳和标注结果配套管理。但真正决定数据能否使用的，是字段、版本、质量记录和使用边界是否清楚。

| 交付要求 | 内容 |
|---|---|
| 可读取 | 字段结构清楚，格式能被系统读取 |
| 可训练 / 可评测 | 能进入训练、评测或审核流程 |
| 可追溯 | 有来源、版本、时间、规则、质量记录 |
| 可解释 | 有标注规范、标签定义、使用说明 |
| 可控使用 | 有权限、合规状态、适用边界和禁用场景 |

对于业务与管理视角的读者，最容易忽略的是“可控使用”。一批标注数据即使格式正确，也可能因为来源授权不清、个人信息未处理、评测集污染、标签规则过期或适用场景不明而不能直接使用。Datasheets for Datasets 提出数据集应附带动机、组成、收集过程和推荐用途等说明；Hugging Face Dataset Cards 也强调数据卡用于帮助用户理解数据内容、偏差和使用语境。这类数据文档不是形式主义，而是在数据进入不同团队、不同模型、不同版本时维持可解释性和可追溯性的基础。

> **实践样例：标准材料中的标注安全和核验要求**  
> 具体材料包括国家标准 GB/T 45674-2025《[网络安全技术 生成式人工智能数据标注安全规范](https://openstd.samr.gov.cn/bzgk/std/newGbInfo?hcno=407584DD0FA2BA19E62E85D3469290B0)》（2025-04-25 发布，2025-11-01 实施）、GB/T 45654-2025《[网络安全技术 生成式人工智能服务安全基本要求](https://std.samr.gov.cn/gb/search/gbDetailed?id=33D40F1160BF5D92E06397BE0A0A5B93)》（2025-04-25 发布，2025-11-01 实施）和 NIST《[AI RMF: Generative AI Profile, NIST AI 600-1](https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.600-1.pdf)》（2024-07）。GB/T 45674-2025 明确把生成式 AI 数据标注安全作为专项对象，GB/T 45654-2025 把生成式 AI 服务安全要求放入国家标准体系，NIST AI 600-1 从生成式 AI 风险管理角度讨论治理、映射、测量和管理。  
> 对应生产阶段：数据准备、标注执行、交付验收、上线前风险评估。  
> 能说明的判断：标注交付物不只是文件格式，还需要覆盖标注规则、人员角色、平台或工具安全、核验记录、数据安全和后续风险管理。  
> 边界：标准材料给出的是规范语境和要求框架，不会替企业完成具体任务定义，也不能替代项目级的数据授权、隐私处理、污染检查和验收记录。

### 2.4 标注数据的沉淀与复用价值

标注数据首先服务具体模型任务，但并不是所有数据都只能用一次。高质量评测集、安全样本、错误案例、领域术语、专家标注结果、标注规范、判断标准和边界案例，通常可以在相近模型或相近业务团队之间复用。

可复用价值最高的通常不是某一批普通训练样本，而是能表达判断标准的材料。例如，客服场景中的“投诉升级边界”、金融场景中的“不得承诺收益”、医疗场景中的“不能替代诊断”、企业搜索中的“证据来源必须可追溯”，这些规则一旦被清楚地写入标签体系、评测集和红队样本，就可以成为后续模型和团队的共同资产。

因此，本节关注的是标注数据如何从一次性项目材料沉淀为可复用资产；至于复用前的权限、污染、版本、追溯和禁用边界，应放入第 3.4 节的验收与使用控制中讨论。

## 3. 标注数据的问题、难点与质量控制

本章回答的问题是：标注数据为什么难？常见质量问题是什么？如何控制、验收、追溯和限定使用？

标注数据质量通常不是最后验收时才决定的，而是在任务定义、样本选择、规则制定、试标校准、批量标注、复核仲裁和数据交付过程中逐步形成的。本报告不把标注生产流程展开成项目管理手册，而是按难点来源、质量问题和控制层级来说明。

### 3.1 标注数据为什么难

| 难点来源 | 说明 | 典型表现 |
|---|---|---|
| 定义难 | 业务目标不容易转成清晰标签 | “优质回答”“风险内容”“相关结果”没有可操作边界 |
| 边界难 | 样本存在灰区，不是简单对错 | 投诉与咨询、讽刺与攻击、建议与承诺难区分 |
| 专业难 | 垂类场景需要专家判断 | 医疗、金融、法律、工业流程依赖专业知识 |
| 一致性难 | 不同标注员理解不同 | 同类样本被不同人打出不同标签 |
| 覆盖难 | 长尾、异常、高风险样本难收集 | 极端故障、罕见疾病、小语种、复杂攻击样本不足 |
| 主观性难 | 偏好、安全、质量判断带有主观性 | 不同用户对“有帮助”“不过度拒答”的判断不同 |
| 变化难 | 上线后业务、政策、用户行为会变化 | 旧规则无法覆盖新产品、新政策和新攻击方式 |
| 复用难 | 一个场景的标签不一定适合另一个场景 | 同样的“高风险”标签在客服和金融中含义不同 |

公开研究也支持这一点。关于数据标注质量管理的研究指出，标注质量问题不仅包括标签错误，也包括一致性、偏差和质量管理方法本身的误用。数据质量维度相关综述则把数据质量与模型性能、公平性、稳健性、安全性和可扩展性联系起来。这说明标注质量不是局部的“返工率”问题，而是会传导到模型表现、评测可信度和上线风险。

LLM 场景进一步放大了这些难点。偏好数据不是天然客观标签，不同标注员可能对“简洁”“礼貌”“安全”“直接回答”有不同判断；安全数据常常涉及灰区，过严会造成无效拒答，过松会放大风险；红队样本会随着模型能力、工具使用和攻击方式变化而变化；LLM-as-Judge 可以提高评测规模，但也会带来偏差、稳定性和元评测等问题，不能简单替代人工判断。

### 3.2 常见质量问题

| 质量问题类型 | 表现 | 对模型生产的影响 |
|---|---|---|
| 标签错误 | 标错、漏标、多标 | 直接污染训练信号或评测结果 |
| 口径不一致 | 同类样本判断不统一 | 模型学到不稳定规则，评测无法解释 |
| 覆盖不足 | 长尾、异常、敏感场景缺失 | 上线后在关键风险场景失败 |
| 样本偏差 | 来源、群体、语言、场景不均衡 | 模型对少数场景、低资源语言或特殊群体表现差 |
| 格式与字段问题 | 字段缺失、格式不合规、系统无法读取 | 训练、评测或审计流程中断 |
| 来源与权限问题 | 授权不清、隐私未处理、版权不明 | 数据无法合规使用，后续追责困难 |
| 版本与追溯问题 | 不知道数据来源、标注规则和修改历史 | 线上问题无法回溯，复用时无法判断适用性 |
| 评测污染问题 | 训练数据与评测数据边界不清 | 评测结果高估，版本对比失真 |
| 误用与过期问题 | 数据被用于不适合场景，或业务规则已变化 | 旧数据在新场景中制造错误判断 |

这些问题中，最容易被低估的是评测污染和误用。一个评测集如果被训练过程“看见”，它就不再能代表真实泛化能力；一批安全样本如果只覆盖旧版攻击方式，也不能说明新版本模型已经安全；一套领域标注如果没有记录专家、依据和版本，后续团队很难判断它是知识资产还是历史包袱。

### 3.3 质量控制方法

标注质量控制可以按五个层级理解：规则质量、样本质量、标注质量、交付质量和使用质量。

| 质量层级 | 控制重点 | 常见材料 |
|---|---|---|
| 规则质量 | 标签定义、边界案例、判断标准 | 标注规范、示例库、反例库、仲裁口径 |
| 样本质量 | 来源、覆盖、长尾、代表性、权限 | 数据来源清单、抽样方案、敏感性标记 |
| 标注质量 | 试标校准、抽检复核、一致性检查、专家仲裁 | 试标报告、IAA 记录、复核记录、仲裁记录 |
| 交付质量 | 字段、格式、版本、质量报告 | 数据字典、版本说明、质量报告、变更记录 |
| 使用质量 | 适用边界、复用限制、评测隔离、后续更新 | 数据卡、模型卡、禁用场景、回归测试记录 |

规则质量是第一层。没有清晰规则，后续抽检只能发现局部错误，无法解决系统性分歧。对于开放式回答和安全判断，规则需要包含正例、反例、边界例和处理灰区的说明。

> **实践样例：行为规范作为规则质量的上游材料**  
> 具体材料包括 OpenAI《[Model Spec](https://model-spec.openai.com/2025-12-18.html)》（2025-12-18 版本）和 Anthropic《[Claude’s Constitution](https://www.anthropic.com/constitution)》及其发布说明《[Claude's new constitution](https://www.anthropic.com/news/claude-new-constitution)》（2026-01-22）。Model Spec 把模型应遵循的行为写成可版本化的公开规范，并用 chain of command 区分 root、system、developer、user 等指令层级；Claude’s Constitution 把“safe、ethical、compliant、helpful”等价值和冲突处理方式写成模型训练会使用的上游文本，并在发布说明中明确提到它可参与合成训练数据和响应排序。  
> 对应生产阶段：行为目标定义、后训练、规则样本构造、安全评测口径制定。  
> 能说明的判断：复杂模型的规则质量不能只靠末端抽检，必须在上游形成可讨论、可更新、可引用的行为规范；这些规范会影响示范数据、偏好数据、安全样本和评测判分的一致性。  
> 边界：行为规范不是标注细则本身，也不能证明模型一定按规范行动。它们说明规则口径如何被公开表达，不能替代项目中的标注员培训、试标校准、冲突仲裁和质量复核。

样本质量是第二层。样本不代表真实任务分布，标注再准确也会导致模型偏斜。对于传统模型，样本覆盖通常关注类别均衡、长尾、异常、噪声和场景差异；对于 LLM，样本覆盖还包括任务类型、语言、文化语境、多轮对话、敏感主题、工具调用、越狱攻击、拒答边界和用户反馈。

标注质量是第三层。试标校准可以暴露规则不清；抽检复核可以发现随机错误；一致性检查可以发现标注员理解差异；专家仲裁可以处理复杂争议。需要注意的是，一致性指标本身不是最终目的。对于高度主观或专家性任务，低一致性可能说明任务定义有灰区，也可能说明需要更细的规则、更多专家参与或分层标注。

交付质量是第四层。数据交付不仅是文件交付，还包括字段说明、版本、质量记录、来源、权限、变更历史和使用说明。没有这些记录，数据很难进入后续训练、评测、追溯和复用。

使用质量是第五层。数据进入模型生产后仍然需要边界。训练数据、验证数据和评测数据要隔离；评测集要防污染；安全数据要随着风险更新；线上反馈要经过清洗、隐私处理和重新标注；复用数据要先确认标签含义和适用场景。

### 3.4 验收、追溯与使用边界

标注数据不只是看当前能不能训练，还要看后续能不能解释、复核、追溯和复用。一个完整的验收视角至少包括八项内容：

| 验收问题 | 需要看到的材料 |
|---|---|
| 标注规范是否存在 | 标签定义、边界案例、正反例、更新记录 |
| 质量记录是否完整 | 抽检比例、错误类型、返工记录、一致性记录 |
| 数据版本是否清楚 | 版本号、生成时间、变更说明、使用模型或规则 |
| 来源和权限是否清楚 | 数据来源、授权状态、隐私处理、版权状态 |
| 训练集和评测集是否隔离 | 数据划分记录、污染检查、评测集封存机制 |
| 是否说明适用场景和禁用场景 | 数据卡、使用说明、风险边界 |
| 是否有错误样本和返工记录 | 错误库、仲裁结论、规则更新说明 |
| 是否能被后续团队理解和复核 | 数据字典、标注说明、负责人和审计线索 |

本节重点不是说明哪些数据有复用价值，而是说明标注数据交付后如何验收、追溯和限定使用。来源、权限、版本、质量记录、评测隔离、污染检查、使用边界、禁用场景和后续更新，决定了数据能否在模型生产中被安全、稳定和可解释地使用。

> **实践样例：模型卡 / 系统卡对上游标注数据的反向要求**  
> 具体材料包括 Mitchell et al.《[Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993)》（arXiv 2018 / FAT* 2019）、OpenAI《[GPT-4o System Card](https://openai.com/index/gpt-4o-system-card/)》（2024-08-08）、Anthropic [Model system cards](https://www.anthropic.com/system-cards) 页面和 Google DeepMind [model cards](https://deepmind.google/models/model-cards/) 页面。Model Cards 论文提出模型发布应说明预期用途、评测条件和限制；GPT-4o System Card 展示了能力、限制、安全评测和第三方评估如何被组织到同一报告中；Anthropic 与 Google DeepMind 的模型卡页面则按模型版本持续列示系统卡或模型卡。  
> 对应生产阶段：模型发布、上线前验收、上线后版本更新和外部说明。  
> 能说明的判断：模型卡 / 系统卡要求上游能够回答“数据来自哪里、如何评测、限制是什么、风险如何缓解、哪些场景不适用”。因此，标注数据在交付时需要保留来源、版本、质量记录、适用边界、评测隔离和风险说明，否则下游文档难以自洽。  
> 边界：模型卡和系统卡通常不会披露完整训练数据、评测集明细或内部阈值。它们能反向说明上游数据记录的重要性，但不能作为内部数据质量已经充分的证明。

### 3.5 质量监控与改进的前沿实践

前三节说明了质量难点、常见问题、五层控制框架和验收边界。本节补充工业与学术界在“如何监控质量”和“如何提高质量”上的更具体做法，作为五层框架的落地参考。这些方法不替代规则和人工判断，而是让监控更可量化、让复核更有重点、让改进更可迭代。

监控的重点是把主观判断转成可量化、可追踪的信号。对于分类、抽取、搜索相关性、视觉检测和开放式回答等不同任务，指标不会完全相同，但都需要回答三个问题：标注员之间是否一致，机器筛查能否优先发现疑似错误，质量变化能否被持续追踪。

| 监控手段 | 做法 | 作用 |
|---|---|---|
| 量化一致性指标 | 使用 Cohen's / Fleiss' Kappa、Krippendorff's α；空间标注（框 / 掩码 / 关键点）使用 IoU 一致性；按标注员、任务、批次持续追踪 | 把“口径不一致”变成可预警信号；阈值需要按任务难度设定，不能机械套用 |
| 黄金集与校准 | 上岗前在校准集上与黄金标准对齐，生产期做重叠抽样和滚动一致性计算，事后把低一致样本送专家仲裁 | 在上岗、生产和事后三个阶段持续核验，及时发现标注漂移 |
| 自动标签错误检测 | 用模型置信度、样本相似性和置信学习给样本打“标签质量分”，排序后优先送人复核 | 压缩人工核验范围，优先暴露最可能错标或最值得复查的样本 |
| 规模化机器质检 | 使用 LLM-as-Judge 或多模型陪审在行级、文档级、对话级评分过滤，对被标红样本再做人审 | 扩大评测与筛查规模，但需要控制偏置、提示敏感和偏好泄漏 |
| 质量趋势看板 | 按批次追踪错误率、返工率、一致性、覆盖率、争议样本类型、上线后失败类型 | 把质量问题从一次性验收变成持续监控，支持版本间对比 |

提高的重点是把发现的问题转回上游，形成可迭代的闭环。单纯增加抽检比例并不总能提高质量；如果规则本身含混，更多抽检只会发现更多分歧。更有效的做法通常是从错误样本反推规则缺口，再更新指南、示例、模型预标注策略和复核机制。

| 改进手段 | 做法 | 作用 |
|---|---|---|
| 规则与指南迭代 | 用可视化示例、决策树、边界案例和反例完善指南；一致性下降时优先检查规则是否有歧义 | 从源头减少分歧，回报通常高于单纯增加抽检层 |
| 主动学习 / 不确定性采样 | 用模型不确定性、样本多样性和错误风险优先挑选最该标注或复核的样本 | 把人力投到信息量最高、最容易出错或最影响模型边界的样本上 |
| 模型预标注 + 人工校正 | AI 先标、人工精修，或由机器给出候选标签和依据后由人确认 | 在保证质量的前提下降低标注耗时，尤其适合规则明确和重复度高的任务 |
| 共识与聚合 | 主观或众包任务用 Dawid-Skene、MACE 等聚合模型替代简单多数投票 | 从噪声标注中更稳健地估计真值，并识别低可靠标注员或低质量任务 |
| 错误回流 | 将线上失败、评测失败、红队样本和用户反馈做错误归因，再更新规则、评测集、训练样本和回归测试集 | 让质量随版本迭代持续提升，并与监控指标形成闭环 |

需要强调的是，这些自动化方法都不取代人工和专家判断。一致性指标只能提示分歧，不能自动判定谁对谁错；自动错误检测和机器质检会引入自身偏差，常见做法是“机器筛查 + 人工兜底”，而不是完全替代人审。对于医疗、金融、法律、安全等高风险任务，机器质检更适合作为预警和分流工具，最终判断仍需要可追溯的人工或专家责任链。

> **实践样例：自动标签错误检测与机器质检的人机协作**  
> 具体材料包括 Northcutt et al.《[Pervasive Label Errors in Test Sets Destabilize Machine Learning Benchmarks](https://arxiv.org/abs/2103.14749)》（NeurIPS Datasets and Benchmarks 2021）、Northcutt et al.《[Confident Learning: Estimating Uncertainty in Dataset Labels](https://arxiv.org/abs/1911.00068)》（JAIR 2021）、Tan et al.《[Large Language Models for Data Annotation and Synthesis: A Survey](https://arxiv.org/abs/2402.13446)》（EMNLP 2024）和 Hovy et al.《[Learning Whom to Trust with MACE](https://aclanthology.org/N13-1132/)》（NAACL 2013）。前两项材料说明基于模型置信度的标签错误检测可以在大规模数据集中优先筛出疑似错标样本；LLM 标注综述说明大模型可参与标注生成、标注评估和标注利用；MACE 代表了从多名标注者噪声中估计可靠标注和真实标签的聚合思路。  
> 对应生产阶段：标注执行、批量质检、评测集维护、反馈回流和版本迭代。  
> 能说明的判断：质量监控可以从人工抽检扩展为“指标预警、机器筛查、专家复核、规则更新”的组合流程；自动化工具可以放大人工判断的覆盖范围，使复核更有重点。  
> 边界：自动错误检测依赖模型和数据分布，LLM-as-Judge 可能受提示、偏好和自信错误影响，聚合模型也不能替代任务定义。因此，这些方法适合作为质量管理工具链的一部分，不能单独作为数据质量充分的证明。

## 4. 公开实践材料的归纳与边界

前文已经在生产阶段、标注数据类型、交付边界和质量控制中穿插使用公开实践材料。本章不再重复展开机构材料，而是归纳这些材料共同呈现出的变化，并说明公开资料能够支持的判断边界。

### 4.1 公开材料呈现出的跨阶段变化

公开材料呈现出的共同变化，不是某一家机构采用了某个单点方法，而是人的判断在模型生产中的位置发生了上移和延展：从逐条标签，扩展到规则设计、样本筛选、偏好判断、风险评测、上线边界和版本维护；从一次性交付，扩展到训练、后训练、评测、部署和反馈中的持续质量控制。对应地，标注数据也从训练样本扩展为行为规范、评测集、红队样本、风险标签、反馈样本和文档化说明。

这一变化使标注工作从“数据生产环节”转向“模型质量形成机制”的一部分。传统标签仍然重要，但它已经不能覆盖全部标注形态。对于 LLM 和多模态模型，后训练中的示范、偏好、安全和规则数据会影响模型行为；评测和红队数据会影响上线判断；模型卡、系统卡和数据卡会反向要求训练与评测材料具备来源、版本、质量记录和适用边界。

### 4.2 公开材料能够说明什么

公开材料能够说明标注、后训练、评测、安全、红队、模型文档化和标准化在模型生产中的位置。它们可以帮助判断前沿机构在哪些环节公开发力，也能说明标注数据已经从传统训练样本扩展为行为规则、评测集、风险样本、反馈材料和追溯文档。

这些材料还能够说明不同类型的公开文件承担不同功能：system card / model card 主要呈现模型能力、限制、评测和部署边界；行为规范说明模型回答、拒答和处理冲突指令的规则来源；技术报告说明后训练、偏好数据、推理训练或自动化评测的技术路径；评测基准和红队材料说明上线验收和风险发现方式；标准与治理文件说明训练数据安全、标注安全、评估评测和责任边界的外部要求。

### 4.3 公开材料不能说明什么

公开材料不能还原完整训练数据、内部标注规范、标注员组织方式、样本加权、返工率、成本结构、供应商关系和内部审计细节。system card / model card 通常公开摘要、评测类别和部分结果，不能还原完整内部决策；论文说明的是技术路径，也不应直接等同于生产实践。

因此，本报告只基于公开材料归纳标注和质量控制在模型生产中的位置，不对机构内部流程作额外推断。对于企业自身模型生产而言，公开材料更适合作为流程设计、质量控制和文档化要求的参考，而不是直接复制为内部标注流程。

## 5. 总结与参考来源

### 5.1 核心总结

第一，标注广泛参与模型生产全周期，但不同阶段参与方式不同。在任务目标定义阶段，标注帮助形成标签、规则和成功标准；在数据准备阶段，标注帮助筛选、清洗、分层和治理数据；在预训练和 mid-training 阶段，标注更多表现为质量标记、领域材料和专家知识；在后训练阶段，标注通过指令、示范、偏好、安全和行为规范直接塑造模型行为；在评测和上线阶段，标注转化为评测集、红队样本、验收材料和风险分类；在反馈迭代阶段，标注又回到错误归因、失败案例和回归测试集中。

第二，标注数据通过不同类型支持模型识别、生成、对齐、安全、评测和迭代能力。任务标签数据支持分类、识别、抽取和排序；生成示范数据支持问答、生成和任务执行；偏好数据支持输出排序和用户体验；安全数据支持拒答、风险识别和红队；评测与反馈数据支持上线验收、版本对比和持续迭代；领域知识数据支持垂类能力和专业边界。不同数据类型之间不是孤立关系，而是共同构成模型能力建设材料。

第三，标注数据质量不只取决于标签准确性，还取决于规则、样本、交付、追溯和使用边界。质量问题包括标签错误、口径不一致、覆盖不足、样本偏差、格式字段问题、来源权限问题、版本追溯问题、评测污染、误用和过期。质量控制也不能只靠最后抽检，而需要覆盖规则质量、样本质量、标注质量、交付质量和使用质量。

第四，前沿公开实践可以作为理解标注工作的现实样例，但不能直接等同于机构内部流程。公开材料显示，人的判断正在从逐条标签扩展到规则设计、偏好判断、安全评测、红队测试、文档化说明和版本维护；但公开材料通常只披露摘要、类别和部分结果，不能还原完整标注流程、样本权重、返工率、成本结构和供应链组织。

第五，质量监控与改进正在从人工抽检扩展为人机协作的持续闭环。一致性指标、黄金集、自动标签错误检测、LLM 辅助质检、主动学习、模型预标注和错误回流，可以让复核更有重点、让质量趋势更可追踪、让规则更新更有依据。但这些方法的边界也很明确：机器筛查只能放大和分流人工判断，不能替代任务定义、专家仲裁和责任追溯。
