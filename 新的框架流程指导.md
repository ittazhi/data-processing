# 标注在模型生产全周期中的参与：调研报告框架流程指导

> 本文件用于指导后续撰写调研报告。它不是正式正文，而是写作框架、取舍原则、调研方法和成文规范。报告目标是：框架清晰、内容完整、重点突出、引用权威，不做实证分析，不写成技术白皮书，也不写成行业宣传稿。

---

## 0. 报告定位

### 0.1 报告主题

本报告讨论**标注在模型生产全周期中的参与方式**，重点关注：

- 标注在模型生产不同阶段的位置；
- 标注数据如何支持模型能力建设；
- 标注数据为什么难、常见问题是什么；
- 如何理解标注数据质量控制、验收、追溯与使用边界；
- 前沿实验室或机构公开实践中，标注、后训练、人工评测、红队、安全评测、自动化辅助等工作如何出现。

报告不做模型实验、不做供应商横评、不做算法细节展开。

### 0.2 写作对象

默认读者是 PM / 管理视角读者，不是算法论文审稿人。正文应做到：

- 主线清楚；
- 术语必要但不过量；
- 技术概念点到为止；
- 重要结论有来源支撑；
- 能解释“为什么重要”，而不是只列“有哪些”。

### 0.3 关于“模型”的范围

报告中的“模型”不只指通用 LLM，也包括：

- 通用 LLM；
- 垂类 LLM；
- 分类模型；
- 推荐 / 排序模型；
- 搜索模型；
- 视觉模型；
- 语音模型；
- NLP 模型；
- 多模态模型；
- 其他企业生产环境中的 AI / 机器学习模型。

注意：报告可以更多使用 LLM 案例，因为当前标注形态变化最明显的场景集中在 LLM 的后训练、偏好、安全和评测环节。但框架不能写成“只适用于 LLM”。

### 0.4 关于“标注”的范围

本报告中的“标注”不只包括传统打标签，还包括：

- 分类、抽取、框选、分割、转写；
- 数据质量标记、来源标记、领域标记、敏感性标记；
- 指令样例、标准答案、改写样本；
- 偏好排序、人工打分、A/B 选择；
- 安全样本、拒答样本、风险标签、红队样本；
- 人工评测、错误归因、线上反馈样本整理；
- 领域术语、事实、规则、流程、证据来源等专家标注。

## 1. 总体写作原则

### 1.1 内容完整，但不逐项摊开

报告必须覆盖完整，但完整不等于每个点都独立成章。处理原则：

- **重要内容展开写**：模型生产阶段、标注数据能力作用、数据质量问题与控制。
- **中等内容合并写**：不同模型差异、团队复用、数据格式、交付边界、责任分工。
- **次要内容带过写**：工具平台功能、供应商管理、成本测算、复杂算法细节。
- **细节内容放附录**：术语表、格式样例、标注规范模板、质量检查清单、参考来源扩展表。

核心思路：

> 内容都要覆盖到，但不重要的内容要通过归类、打包、概括和附录处理，不能让正文变成材料仓库。

### 1.2 不写“管理建议”大章

本报告定位是调研框架和现状梳理，不主动设置“管理建议”章节。

原因：管理建议容易变成正确废话，例如“建立机制、加强质量、持续迭代、数据资产化”。如果老板后续要求，可以单独补；本报告正文以**总结与参考来源**收束即可。

### 1.3 前置结论不宜写太重

开头可以放简短摘要，但不需要一开始写详细结论。

建议开头摘要控制在 1 段：

> 本报告围绕标注在模型生产全周期中的参与方式展开，重点分析标注数据如何进入数据准备、训练、后训练、评测、安全和反馈迭代环节，以及不同标注数据如何支持模型能力建设。报告不做实证分析，不展开算法细节，重点梳理流程、数据、质量和前沿实践观察。

详细总结放最后。

## 2. 最终框架汇总与推荐主框架

建议使用以下框架，暂记为：

# 框架 04：分阶段模型生产与标注数据质量导向框架

```text
0. 报告口径与范围说明
   0.1 调研对象与基本范围
   0.2 本报告不覆盖的内容

1. 模型生产阶段与标注参与位置
   1.1 分阶段模型生产流程
   1.2 标注在不同阶段的参与方式
   1.3 不同模型在同一生产阶段中的差异提示

2. 标注数据如何支持模型能力建设
   2.1 标注数据的形成：载体、动作与产物
   2.2 关键标注数据类型及其支持的模型能力
   2.3 数据格式、交付物与使用边界
   2.4 标注数据的沉淀与复用价值

3. 标注数据的问题、难点与质量控制
   3.1 标注数据为什么难
   3.2 常见质量问题
   3.3 质量控制方法
   3.4 验收、追溯与使用边界

4. 前沿实践观察：标注、评测与质量把控
   4.1 前沿实践调研口径与材料来源
   4.2 实际调研流程
   4.3 前沿材料筛选标准
   4.4 证据抽取模板
   4.5 归纳方式
   4.6 正文写法：行业观察框
   4.7 注意边界

5. 总结与参考来源
   5.1 核心总结
   5.2 参考来源分类
```

### 2.1 最终框架汇总（交付版）

最终报告建议采用“**分阶段模型生产 + 标注数据能力作用 + 质量问题控制 + 前沿公开实践观察**”四段式主线。它既保留完整覆盖，也避免把正文写成标注项目执行手册或公司案例汇编。

```text
定范围
↓
按模型生产阶段定位标注参与
↓
说明标注数据如何支持模型能力
↓
分析标注数据难点、质量问题与控制方法
↓
基于实际调研归纳前沿公开实践
↓
总结与来源归档
```

| 章节 | 核心问题 | 重点抓 | 轻轻带过 | 建议呈现形式 |
|---|---|---|---|---|
| 0. 报告口径与范围说明 | 本报告讨论什么、不讨论什么？ | 模型范围、标注范围、不做实证分析、不做工具横评 | 术语铺垫 | 短段落为主 |
| 1. 模型生产阶段与标注参与位置 | 标注在模型生产哪些阶段出现？ | 分阶段流程、后训练、评测、安全、反馈；说明参与方式而不是讲训练技术 | 不同模型的完整横向对比 | 流程图 + 阶段说明表 |
| 2. 标注数据如何支持模型能力建设 | 不同标注数据支持什么能力？ | 数据类型、能力对应、交付边界、复用价值 | 具体文件格式细节 | 核心对应表 + 少量解释文字 |
| 3. 标注数据的问题、难点与质量控制 | 标注数据为什么难，如何保证可用？ | 难点来源、质量问题、质量控制、追溯与使用边界 | 完整标注项目执行流程 | 文字解释 + 问题/控制表 |
| 4. 前沿实践观察 | 前沿实验室和机构公开材料中，哪些实践与标注、评测、后训练、质量把控有关？ | 最新公开成果、实际实践、机构讨论、质量把控位置、边界；中美兼顾，成果与实践兼顾 | 公司排名、主观评优、预设趋势 | 行业观察框 + 证据地图 |
| 5. 总结与参考来源 | 报告最后如何收束？ | 总结标注全周期参与、能力作用、质量控制和前沿实践观察；参考来源分类列出 | 管理建议大章 | 短总结 + 来源分类 |

第 4 章不是预设“前沿实践必然是什么”，而是把实际调研结果放入框架中。若某一类实践公开材料不足，应写“公开材料不足以判断”，不要硬补。

## 3. 各章节写作指导

## 0. 报告口径与范围说明

### 目标

用最短篇幅讲清楚：报告讨论什么、不讨论什么、模型和标注的范围是什么。

### 建议写法

不要拆太多小标题，正文半页以内即可。

建议包括：

- 调研对象：标注在模型生产全周期中的参与；
- 模型范围：不只通用 LLM，也包括垂类 LLM 和其他模型；
- 标注范围：不只传统打标签，也包括指令、偏好、安全、评测、反馈等；
- 不覆盖内容：不做实证分析、不展开算法细节、不做供应商报价和工具横评。

### 需要避免

- 一开头就写很重的三大结论；
- 一开头就堆术语；
- 把报告写成大模型技术科普。

## 1. 模型生产阶段与标注参与位置

### 目标

回答：**标注到底在模型生产哪些阶段出现？哪些阶段只是辅助，哪些阶段会直接影响模型行为、能力和上线判断？**

### 1.1 分阶段模型生产流程

不要只写：

```text
数据准备 → 训练 → 评测 → 上线
```

这太粗，会把后续内容涉及的关键环节全部压扁。

建议使用：

```text
任务目标与成功标准定义
↓
数据规划与准备
↓
基础能力训练 / 预训练
↓
领域或能力增强 / Mid-training / 持续训练
↓
任务适配与行为校准 / Post-training / 后训练
↓
评测、安全审查与上线验收
↓
部署上线与运行监控
↓
反馈回流与版本迭代
```

说明：

- 使用“通用名称 + LLM 常用术语括注”，避免框架只适用于 LLM；
- pre-training、mid-training、post-training 不要写成所有模型都严格具备的固定步骤；
- 这些阶段用于帮助理解标注参与位置，而不是展开训练技术细节。

### 1.2 标注在不同阶段的参与方式

不建议使用“深 / 中 / 轻”作为正式表述，容易被误解成价值高低。

建议改成三类：

| 参与类型 | 含义 | 典型位置 |
|---|---|---|
| 核心塑形型参与 | 直接影响模型行为、输出标准、安全边界或上线判断 | 任务目标定义、后训练、人工评测、安全审查、错误样本回流 |
| 能力支撑型参与 | 支持特定能力、领域能力、数据筛选和评测集建设 | 数据规划、领域增强 / Mid-training、评测集建设、反馈迭代 |
| 治理辅助型参与 | 服务数据治理、清洗、过滤、追溯和风险控制 | 预训练数据处理、来源标记、敏感性标记、格式治理 |

判定依据不是标注人力多少，也不是数据量大小，而是：

- 是否直接进入训练 / 后训练 / 评测核心数据；
- 是否定义模型应该如何回答；
- 是否影响上线判断；
- 是否需要高密度人工判断；
- 是否影响后续版本迭代。

### 1.3 不同模型在同一生产阶段中的差异提示

注意：不要在每个阶段机械比较通用 LLM、垂类 LLM、其他模型。那会偏离主题，变成模型类型对比报告。

正确处理方式：

> 阶段是主线，标注是重点，模型差异只是校准项。

只有当模型差异影响以下内容时，才讨论：

- 标注对象；
- 标注数据类型；
- 质量标准；
- 评测口径；
- 领域专家参与程度；
- 安全和合规边界。

示例写法：

> 本报告不为不同模型分别建立流程，而是在同一生产阶段中说明差异。只有当模型类型影响标注对象、质量标准或评测口径时，才单独提示。例如，通用 LLM 在后训练阶段更关注通用指令、偏好和安全数据；垂类 LLM 更关注领域术语、专业事实、业务流程和专家判断。

## 2. 标注数据如何支持模型能力建设

### 目标

回答：**不同标注数据到底支持模型什么能力？它们只服务单次训练，还是可以沉淀为后续能力建设材料？**

### 2.1 标注数据的形成：载体、动作与产物

这一节要概括，不要摊开成清单。

#### 原始载体

可一句带过：

> 标注数据的原始载体包括文本、图像、音频、视频、日志和结构化数据。不同载体上的标注动作不同，但最终都需要转化为可训练、可评测、可追溯的数据材料。

#### 标注动作分类

建议采用六类，完整但不碎：

| 标注动作类别 | 覆盖内容 |
|---|---|
| 判断类 | 分类、打分、排序、审核、风险判断 |
| 定位 / 抽取类 | 框选、分割、实体抽取、关系抽取、时间片段定位 |
| 转写 / 结构化类 | 语音转写、OCR 校正、字段整理、表格结构化 |
| 关联 / 对齐类 | 图文匹配、证据对齐、问答来源绑定、视频文本对齐 |
| 生成 / 改写类 | 编写指令、标准答案、改写样本、示范回答 |
| 评测 / 反馈类 | 人工评测、错误归因、用户反馈整理、失败案例标注 |

### 2.2 关键标注数据类型及其支持的模型能力

这是本章核心。建议用表格。

| 标注数据类型 | 主要内容 | 支持能力 |
|---|---|---|
| 数据治理 / 质量标记数据 | 来源、语种、领域、质量等级、敏感性、可用性 | 数据筛选、风险过滤、训练数据治理 |
| 任务标签数据 | 分类、实体、关系、目标框、排序标签 | 分类、识别、抽取、检测、排序 |
| 生成示范数据 | 指令、标准答案、改写样本、示范回答 | 生成、问答、任务执行 |
| 偏好数据 | 回答排序、A/B 选择、人工打分 | 偏好对齐、输出排序、用户体验 |
| 安全数据 | 风险标签、拒答样本、违规样本、红队样本 | 安全控制、拒答能力、风险识别 |
| 评测与反馈数据 | 人工评测、错误样本、用户反馈、回归测试样本 | 上线验收、版本对比、持续迭代 |
| 领域知识数据 | 术语、事实、规则、流程、专家判断、证据来源 | 垂类能力、专业能力、知识边界 |

注意：不要再把“证据来源、引用依据、事实边界”单独拆成新类别，容易碎。放入“领域知识数据”即可。

### 2.3 数据格式、交付物与使用边界

这一节不要按文件后缀展开。重点不是 CSV、JSONL、COCO，而是交付是否能被训练、评测、追溯和使用。

建议按交付要求写：

| 交付要求 | 内容 |
|---|---|
| 可读取 | 字段结构清楚，格式能被系统读取 |
| 可训练 / 可评测 | 能进入训练、评测或审核流程 |
| 可追溯 | 有来源、版本、时间、规则、质量记录 |
| 可解释 | 有标注规范、标签定义、使用说明 |
| 可控使用 | 有权限、合规状态、适用边界和禁用场景 |

格式只需轻写：

> 文本和对话数据常见为 CSV、JSON、JSONL、Parquet；图像检测常见为 COCO、YOLO 等；语音和视频数据通常需要原始文件、时间戳和标注结果配套管理。但格式本身不是核心，核心是字段、版本、质量记录和使用边界是否清楚。

### 2.4 标注数据的沉淀与复用价值

不要单独开“团队复用”章节，但必须覆盖。

建议写法：

> 标注数据首先服务具体模型任务，但高质量评测集、安全样本、错误案例、领域术语和专家标注结果，可以在相近模型或相近业务团队之间复用。复用前需要确认标签含义、数据来源、质量记录和适用边界。

注意：不要写成“标注数据都可以复用”。

更稳妥的判断是：

- 高度绑定单一业务规则的数据，复用前必须重新确认；
- 评测、安全、错误样本、领域知识、标注规范通常更有复用价值；
- 没有来源、版本、质量记录、适用边界的数据，不应直接复用。

## 3. 标注数据的问题、难点与质量控制

### 目标

回答：**标注数据为什么难？常见质量问题是什么？如何控制、验收、追溯和限定使用？**

注意：本报告不单独展开“标注生产流程”。

可以用一句话带过流程：

> 标注数据质量通常不是最后验收时才决定的，而是在任务定义、样本选择、规则制定、试标校准、批量标注、复核仲裁和数据交付过程中逐步形成的。

这样覆盖了流程，但不把报告写成标注项目管理手册。

### 3.1 标注数据为什么难

建议按难点来源写：

| 难点来源 | 说明 |
|---|---|
| 定义难 | 业务目标不容易转成清晰标签 |
| 边界难 | 样本存在灰区，不是简单对错 |
| 专业难 | 垂类场景需要专家判断 |
| 一致性难 | 不同标注员理解不同 |
| 覆盖难 | 长尾、异常、高风险样本难收集 |
| 主观性难 | 偏好、安全、质量判断带有主观性 |
| 变化难 | 上线后业务、政策、用户行为会变化 |
| 复用难 | 一个场景的标签不一定适合另一个场景 |

### 3.2 常见质量问题

建议保留以下分类，不能只写“标签错、不一致、不完整”。

| 质量问题类型 | 表现 |
|---|---|
| 标签错误 | 标错、漏标、多标 |
| 口径不一致 | 同类样本判断不统一 |
| 覆盖不足 | 长尾、异常、敏感场景缺失 |
| 样本偏差 | 来源、群体、语言、场景不均衡 |
| 格式与字段问题 | 字段缺失、格式不合规、系统无法读取 |
| 来源与权限问题 | 授权不清、隐私未处理、版权不明 |
| 版本与追溯问题 | 不知道数据来源、标注规则和修改历史 |
| 评测污染问题 | 训练数据与评测数据边界不清 |
| 误用与过期问题 | 数据被用于不适合场景，或业务规则已变化 |

### 3.3 质量控制方法

不按生产流程写，按质量层级写。

| 质量层级 | 控制重点 |
|---|---|
| 规则质量 | 标签定义、边界案例、判断标准 |
| 样本质量 | 来源、覆盖、长尾、代表性、权限 |
| 标注质量 | 试标校准、抽检复核、一致性检查、专家仲裁 |
| 交付质量 | 字段、格式、版本、质量报告 |
| 使用质量 | 适用边界、复用限制、评测隔离、后续更新 |

这一层分类的好处：覆盖了生产过程，但不把流程喧宾夺主。

### 3.4 验收、追溯与使用边界

需要说明：标注数据不只是看当前能不能训练，还要看后续能不能解释、复核、追溯和复用。

应覆盖：

- 标注规范是否存在；
- 质量记录是否完整；
- 数据版本是否清楚；
- 来源和权限是否清楚；
- 训练集、评测集是否隔离；
- 是否说明适用场景和禁用场景；
- 是否有错误样本和返工记录；
- 是否能被后续团队理解和复核。

## 4. 前沿实践观察：标注、评测与质量把控

### 目标

这一章不是预设“前沿趋势是什么”，而是通过公开材料调研，观察前沿实验室或机构实际在哪些环节发力。

必须遵守：

> 先调研公开材料，再归纳观察；有就写，没有就不写；不能先设定这里应该是 OpenAI、那里应该是 Anthropic、再去找材料硬贴。

### 4.1 前沿实践调研口径与材料来源

前沿实践部分必须坚持：**先调研，再归纳；有就写，没有就不写。**

不能先设定“OpenAI 应该写行为规范、Anthropic 应该写 constitution、DeepMind 应该写自动化红队”这类分工，再去找材料填空。候选机构名单只是检索入口，不是正文目录。最终写什么，由实际公开材料决定。

#### 材料时效

一般原则是**越新越好**，尤其是 system card、model card、安全报告、模型行为规范、红队报告、部署说明、透明度报告、标准讨论等快速变化材料。写作时应重新检索最新版本，并记录发布时间或版本日期。

处理方式：

- 近 12—24 个月材料优先；
- 经典论文或早期工作可以作为背景，但不能替代最新实践；
- 同一机构有新版材料时，以新版为准，旧材料只作历史背景；
- 若材料发布日期、版本或适用模型不清楚，需要标注“不确定”；
- 如果公开材料不足，明确写“公开材料不足以判断”，不要脑补。

#### 成果与实践兼顾

前沿调研不能只看论文成果，也不能只看产品宣传。应同时关注“成果”和“实践”。

| 类型 | 例子 | 用途 |
|---|---|---|
| 成果类材料 | 论文、技术报告、模型卡、系统卡、评测报告、开源模型说明、benchmark 报告 | 判断方法、数据、评测和安全机制如何被公开描述 |
| 实践类材料 | 安全部署说明、红队报告、透明度报告、模型行为规范、产品限制说明、反馈机制说明、上线评测说明 | 判断真实生产或治理环节中如何把控质量、风险和反馈 |
| 讨论类材料 | 标准组织文件、监管 / 治理讨论、论坛材料、机构公开演讲、行业联盟报告 | 判断行业共识、争议和治理关注点 |

正文中不必平均展示三类材料，但调研阶段应尽量都检索。最终是否写入正文，以材料质量和与主线相关性为准。

#### 中、美及国际机构兼顾

前沿实践应兼顾中、美及国际机构的公开材料。不是为了凑数量，而是避免报告只呈现单一地区的实践视角。

可检索对象包括但不限于以下类别，写作时以最新公开材料和机构名称为准：

- 美国及国际前沿实验室 / 机构：OpenAI、Anthropic、Google DeepMind、Meta AI、Microsoft Research / AI Frontiers、xAI、AI2、Stanford CRFM、NIST、Frontier Model Forum、OECD 等；
- 中国前沿模型团队 / 企业：DeepSeek、智谱 AI、百度文心、阿里通义、腾讯混元、字节豆包、月之暗面 Kimi、MiniMax、商汤、阶跃星辰等；
- 中国高校、研究院与公共机构：清华、北大、中科院、上海人工智能实验室、鹏城实验室、中国信通院、标准化组织、行业治理研究机构等；
- 其他国家和地区的公开模型团队、研究机构、标准组织。

注意：以上只是检索范围，不代表都要写入正文。正文只保留能支撑报告判断的材料。机构名单也不应被写成“前沿实践章节目录”。

### 4.2 实际调研流程

建议按以下流程做，避免先预设结论：

```text
确定调研问题
↓
按中、美及国际机构范围检索最新公开材料
↓
记录材料来源、发布时间、版本、材料类型
↓
抽取与标注、评测、后训练、红队、安全、反馈、质量治理相关的信息
↓
判断材料对应模型生产哪个阶段
↓
区分“研究成果”“生产实践”“机构讨论”“政策 / 标准”
↓
判断材料是否能支撑报告主线
↓
归纳观察，而不是按公司堆案例
```

调研时要区分三类材料：

| 类型 | 怎么用 | 注意事项 |
|---|---|---|
| 前沿成果 | 说明方法、数据、评测思路如何发展 | 不把论文结果直接等同于生产实践 |
| 公开实践 | 说明机构在模型发布、安全、评测、后训练中实际披露了什么 | 不超出公开材料推断内部流程 |
| 机构讨论 / 标准 | 说明行业或治理层面对数据质量、风险、评测的共识 | 不用宏观报告替代具体模型实践 |

前沿实践章节最终应回答：**公开材料实际显示，前沿机构在哪些环节发力；这些发力点如何对应标注数据、评测数据、质量控制和使用边界。**

### 4.3 前沿材料筛选标准

每条前沿实践进入正文前，需要过筛：

| 筛选问题 | 保留标准 |
|---|---|
| 是否来自公开、可引用的一手材料？ | 优先官方 system card、model card、研究博客、论文、技术报告、机构报告 |
| 是否足够新？ | 快速变化材料优先使用最新版本；经典材料只作背景或方法来源 |
| 是否能说明标注或人工反馈的作用？ | 必须对应标注数据、行为规范、评测、安全、反馈等主题 |
| 是否说明具体生产环节？ | 最好能对应 pre-training、mid-training、post-training、评测、红队、上线反馈等阶段 |
| 是否兼顾成果和实践？ | 既关注论文 / 报告 / 模型卡，也关注部署、安全、治理、红队、评测实践 |
| 是否覆盖中、美及国际机构公开讨论？ | 不要求数量平均，但不能只看单一国家或单一实验室材料 |
| 是否能支撑报告判断？ | 不能只显得前沿，要能解释为什么质量控制、后训练或自动化重要 |
| 是否避免横向评测？ | 不写“谁更强”，只写“公开实践说明什么” |

### 4.4 证据抽取模板

每个材料建议做成素材卡：

```text
材料名称：
机构 / 实验室：
国家 / 地区：
发布时间 / 版本日期：
材料类型：system card / model card / research blog / paper / policy / safety report / standard / public talk / other
材料性质：成果类 / 实践类 / 治理讨论类
涉及阶段：数据准备 / 预训练 / mid-training / 后训练 / 评测 / 红队 / 部署 / 反馈
涉及标注类型：指令 / 偏好 / 安全 / 评测 / 数据治理 / 领域知识 / 其他
材料中实际说了什么：
能支撑的报告判断：
不能过度解读的地方：
是否为最新公开版本：是 / 否 / 不确定
是否进入正文：是 / 否
```

### 4.5 归纳方式

不要按公司写：

```text
OpenAI 做法
Anthropic 做法
DeepMind 做法
```

除非报告任务明确要求横向对比，否则不要这样写。

更推荐先根据实际材料归纳，再决定使用哪些小节。可能的归纳维度包括但不限于：

- 行为规范如何进入训练和评测；
- 后训练中如何使用指令、偏好、安全数据；
- 人工评测和红队如何参与上线把关；
- 自动化评测、自动化红队如何补充人工评测；
- 数据治理、模型卡、系统卡、数据卡如何帮助追溯；
- 反馈和规则更新如何影响后续数据；
- 中美机构在模型评测、安全治理、数据质量、标准建设上的公开讨论；
- 前沿实践中有哪些没有公开、不能确认的部分。

这些归纳必须来自实际调研结果。调研后发现哪类材料充分，就写哪类；没有充分材料就不写或标注“不足以判断”。

归纳时应注意中、美材料都要尝试覆盖，但不做机械配平。若中国或美国某一侧公开材料较少，可以说明材料限制；不要为了“看起来平衡”硬凑案例。成果材料和实践材料也应兼顾：既看模型发布、模型卡、技术报告，也看后训练、评测、红队、数据治理、安全部署等具体做法。不要为了凑结构强行设置“高价值标注”“自动化辅助”“某机构案例”之类标题。

### 4.6 正文写法：行业观察框

行业材料建议用“行业观察框”，不要写成长案例。

每个观察框包含：

```text
行业观察：材料 / 实践名称

1. 公开材料说了什么；
2. 它对应模型生产哪个阶段；
3. 它说明标注、评测或质量把控中的什么问题；
4. 不能过度解读的边界。
```

每个观察框约 120—200 字。材料复杂时可以稍长，但不要写成公司案例研究。

### 4.7 注意边界

前沿实践部分不能做以下事情：

- 不做 OpenAI / Anthropic / DeepMind 横向排名；
- 不写“某公司一定采用了某方法”，除非公开材料明确说明；
- 不把论文方法等同于生产实践；
- 不用二手媒体替代官方材料；
- 不因为某公司没公开就推断其没有做；
- 不把“自动化辅助”写成“自动替代人工”；
- 不把“安全拒答更多”写成“安全更好”；
- 不把“有 system card / model card”写成“完全透明”。

## 5. 总结与参考来源

### 5.1 核心总结

总结不写管理建议。建议围绕 3—4 条：

1. 标注广泛参与模型生产全周期，但不同阶段参与方式不同；
2. 标注数据通过不同类型支持模型识别、生成、对齐、安全、评测和迭代能力；
3. 标注数据质量不只取决于标签准确性，还取决于规则、样本、交付、追溯和使用边界；
4. 前沿实践应基于公开材料归纳，不预设趋势；可观察到什么写什么，不能确认的部分不强行写。

### 5.2 参考来源分类

参考来源建议分类，而不是堆列表：

| 来源类型 | 用途 |
|---|---|
| 官方 system card / model card | 支撑模型阶段、安全、评测、红队、上线限制 |
| 模型行为规范 / constitution / policy | 支撑行为规则、后训练、评测口径 |
| 研究论文 / 技术博客 | 支撑方法背景，例如偏好数据、后训练、自动化评测 |
| 数据治理 / 风险框架 | 支撑质量、风险、追溯、责任边界 |
| 标注平台 / 数据格式文档 | 只作辅助，支撑格式、工具能力，不作前沿实践主证据 |

引用原则：

- 关键判断必须有权威来源；
- 事实性内容要标明来源和日期；
- 前沿实践优先使用较新材料，写作时必须重新检索；
- 中、美机构材料都应主动检索，最终是否纳入取决于公开性、相关性和权威性；
- 成果材料与实践材料都要考虑，不能只看发布结果，也不能只看方法讨论；
- 不引用营销软文支撑核心判断；
- 来源不足时明确写“公开材料不足以判断”。

## 6. 表格、文字与行业观察的比例控制

### 6.1 不机械限制表格数量

表格不是越少越好，也不是越多越专业。判断标准是：**什么形式更清楚，就用什么形式。**

不能为了去 AI 感强行压缩表格。随意压缩会导致读者不明所以。

### 6.2 什么适合用表格

适合表格：

- 分类关系；
- 对应关系；
- 阶段差异；
- 问题—方法—边界；
- 数据类型—能力关系；
- 质量问题—控制方法。

### 6.3 什么适合用文字

适合文字：

- 因果解释；
- 判断依据；
- 章节过渡；
- 行业实践的启发；
- 不确定性说明；
- 为什么某个内容重要。

### 6.4 什么适合用观察框

行业实践、前沿实验室公开材料、典型做法适合用观察框。

观察框的作用是让报告更具体、更生动，但不能喧宾夺主。

### 6.5 建议版面节奏

每章可以采用：

```text
短段落说明本章问题
↓
必要的流程图或核心表格
↓
文字解释关键判断
↓
行业观察框（如有）
↓
一段小结
```

避免连续多张表，也避免连续大段文字。

### 6.6 建议比例

不是硬性规定，只作参考：

| 内容形态 | 建议占比 | 作用 |
|---|---:|---|
| 文字解释 | 50%—60% | 负责逻辑、判断、过渡 |
| 表格 / 流程图 | 25%—35% | 负责分类、对应关系、阶段差异 |
| 行业观察框 | 10%—15% | 负责前沿感和具体性 |
| 引用来源 | 适量 | 支撑关键判断，不喧宾夺主 |

## 7. 可读性与去 AI 感写法

### 7.1 少用口号式句子

避免频繁使用：

- 赋能；
- 抓手；
- 底座；
- 全链路；
- 多维协同；
- 闭环生态；
- 关键基础设施；
- 系统性工程。

这些词不是完全不能用，但不能密集出现。

### 7.2 用具体场景替代抽象堆叠

不要只写：

> 标注数据支持模型评测、安全和持续迭代。

可以写：

> 例如客服模型上线后，如果频繁在退款、投诉、升级工单中回答错误，这些失败案例就可以被回收、归因、重新标注，形成下一轮评测和训练数据。

### 7.3 每章只回答一个核心问题

建议每章开头明确本章问题：

- 第 1 章：标注在模型生产哪些阶段参与？
- 第 2 章：标注数据支持什么模型能力？
- 第 3 章：标注数据为什么难、如何控质量？
- 第 4 章：前沿公开实践中，哪些做法与标注、评测和质量把控有关？

### 7.4 不要把表格当正文

表格用于清楚呈现分类和对应关系，不能代替解释。

每张表前后都应有文字说明：

- 表格为什么出现；
- 读者应该看什么；
- 表格说明了什么判断。

### 7.5 不要强行完整到排版灾难

完整覆盖靠分类和打包，不靠把所有内容逐项写长。

例如：

- 标注动作压成六类；
- 标注数据类型压成七类；
- 模型能力压成六类；
- 质量控制压成五层；
- 前沿实践按实际调研结果归纳，不预设目录。

## 8. 最终检查清单

写完报告前，用以下清单检查。

### 8.1 逻辑检查

- [ ] 是否先说明范围，再进入模型生产阶段？
- [ ] 是否按阶段说明标注参与位置？
- [ ] 是否解释标注数据如何支持模型能力？
- [ ] 是否说明标注数据难点、质量问题和控制方法？
- [ ] 前沿实践是否由实际调研归纳，而不是预设结论？
- [ ] 是否避免把报告写成工具介绍、公司对比或算法科普？

### 8.2 内容覆盖检查

- [ ] 是否覆盖传统标注和 LLM 后训练相关标注？
- [ ] 是否覆盖 pre-training、mid-training、post-training、评测、反馈等阶段？
- [ ] 是否覆盖数据治理 / 质量标记数据？
- [ ] 是否覆盖偏好、安全、评测、反馈、领域知识数据？
- [ ] 是否覆盖格式、交付物、追溯、使用边界？
- [ ] 是否覆盖质量问题中的来源、权限、评测污染、误用、过期？
- [ ] 是否覆盖中、美及国际前沿实验室或机构公开实践，但不做横向评测？
- [ ] 前沿实践是否兼顾成果类材料和实践 / 治理讨论类材料？
